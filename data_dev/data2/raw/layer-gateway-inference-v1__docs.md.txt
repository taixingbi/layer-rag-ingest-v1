# Layer Gateway Reference

This document is a quick technical reference for `layer-gateway-inference-v1`.

## 1) What this service does

The gateway accepts OpenAI-compatible chat requests and routes them to healthy vLLM backends.

Primary route:

- `POST /v1/chat/completions`

Operational routes:

- `GET /healthz`
- `GET /metrics`

## 2) High-level request flow

1. Request arrives at `routes.py`.
2. Gateway validates payload and assigns/requests correlation IDs.
3. Request is classified (small/medium/large/streaming_long).
4. Request enters `AdmissionQueue`.
5. Scheduler picks an eligible backend using score + health policies.
6. Proxy forwards request to selected backend (with retry policy).
7. Runtime health/latency signals are updated.
8. Structured logs and Prometheus metrics are emitted.

```mermaid
flowchart LR
    C[Client] --> A[API Routes]
    A --> Q[AdmissionQueue]
    Q --> S[Scheduler]
    S --> R[Backend Registry]
    S --> P[Proxy Client]
    P --> B1[vLLM Backend 1]
    P --> B2[vLLM Backend 2]
    A --> L[Structured Logs]
    A --> M[Prometheus Metrics]
```

## 3) Core modules

- `app/api/routes.py`: API handlers, request validation, queue admission.
- `app/queue/admission_queue.py`: bounded queue + queue gauges.
- `app/scheduler/dispatcher.py`: dispatch loop and scheduling decisions.
- `app/scheduler/scoring.py`: backend score function (inflight/latency/error/hot/overload).
- `app/proxy/client.py`: upstream HTTP call path, retries, streaming handling.
- `app/backends/registry.py`: backend state registry and eligibility checks.
- `app/backends/health.py`: circuit-breaker transitions (`closed`, `open`, `half_open`).
- `app/core/logging.py`: structured JSON logging format + gateway event logger.
- `app/metrics/prometheus.py`: metric definitions and helpers.
- `app/metrics/sync.py`: sync in-memory backend state to Prometheus gauges.

## 4) Important runtime concepts

- **AdmissionQueue**: bounded staging queue before dispatch; protects system under burst.
- **Request class**: estimated work size from request body; used in routing fairness.
- **Circuit breaker**:
  - `closed`: normal traffic
  - `open`: backend temporarily blocked
  - `half_open`: limited probe traffic after cooldown
- **drained**: administrative flag to stop new scheduling to a backend.
- **recent_error_rate**: EWMA failure signal for eligibility/scoring.

## 5) Config usage

Config is loaded by `load_gateway_config()` from:

1. explicit path argument (if passed),
2. `GATEWAY_CONFIG` env var,
3. default `config.yaml`.

Main config sections:

- `server`: bind host/port, request timeout.
- `scheduler`: queue sizing and dispatch cadence.
- `routing`: scoring weights and anti-hotspot controls.
- `backends`: upstream endpoints and concurrency limits.
- `health`: circuit-breaker thresholds.
- `retry`: retry attempts and retryable status codes.

## 6) Logging and tracing IDs

Structured logs are emitted as one JSON line per event.

Gateway event fields include:

- `event`, `service`, `env`, `trace_id`, `request_id`, `session_id`,
- optional `backend`, `latency_ms`, `status_code`, and `error`.

ID behavior:

- `request_id`: from `x-request-id` or generated by gateway.
- `trace_id`: from `x-trace-id` if provided by caller.
- `session_id`: from `x-session-id` if provided by caller.

```mermaid
flowchart TD
    H[Incoming headers] --> X1{x-request-id present?}
    X1 -- yes --> RID[use x-request-id]
    X1 -- no --> NEWRID[generate request_id]
    H --> X2{x-trace-id present?}
    X2 -- yes --> TID[use trace_id]
    X2 -- no --> TID0[trace_id = '-']
    H --> X3{x-session-id present?}
    X3 -- yes --> SID[use session_id]
    X3 -- no --> SID0[session_id = '-']
```

## 7) Metrics quick map

Traffic and reliability:

- `gateway_requests_total`
- `gateway_request_errors_total{backend=...}`
- `gateway_request_retries_total`
- `gateway_rejections_total{reason=...}`

Queue pressure:

- `gateway_queue_depth`
- `gateway_queue_age_ms`

Backend state:

- `gateway_backend_inflight{backend=...}`
- `gateway_backend_error_rate{backend=...}`
- `gateway_backend_circuit_state{backend=...}`
- `gateway_backend_ewma_ttft_ms{backend=...}`
- `gateway_backend_ewma_e2e_ms{backend=...}`

Latency:

- `gateway_request_latency_ms` (histogram)

## 8) Common failure patterns

- **Queue full / age exceeded**: gateway returns 503; tune queue/scheduler or reduce load.
- **Transport errors/timeouts**: upstream unreachable or slow; check backend URLs and network path.
- **Frequent retries**: one backend unstable; check `recent_error_rate`, circuit state, and backend logs.
- **Circuit open**: backend failing repeatedly; wait cooldown or drain backend for investigation.
